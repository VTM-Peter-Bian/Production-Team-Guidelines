規範 01

每日工作流程

開發組成員每日到崗、離崗的標準工作節奏與匯報要求。

10:00

到達公司，開始工作

到崗後依序完成以下事項：

  • 在公司門口打卡機刷臉打卡
  • 查看 Outlook 郵件與行事曆，並按 👍 標記已讀（詳見 工作習慣 01）
  • 瀏覽 SharePoint 最新資訊
  • 查看 Teams 訊息，並按 👍 標記已讀（詳見 工作習慣 01）
  • 更新 T&P（Task & Progress，用於管理同事的工作進度）
  • 回覆相關訊息與待辦內容（須標記序號，詳見 工作習慣 03）
13:00 – 14:00

午飯時間

午休時段，暫停工作事項。

17:00 前

每日進度匯報

下班前須完成以下匯報工作：

  • 更新 T&P（Task & Progress）
  • 在 Planner 對應的 Task 上進行每日工作進度匯報
  • 附上支撐材料，例如：
    • APK 安裝包
    • 截圖 / 圖片
    • 錄影 / 影片
18:00 前

代碼提交

使用 GitHub Desktop 將當日代碼更變提交至 GitHub，詳見 GitHub Desktop 提交規範。

18:00

下班

完成當日工作後，依序完成以下事項再離開公司：

序號 事項 說明
1 關閉設備 關閉個人電腦及顯示屏
2 刷臉打卡 離開前在公司門口打卡機刷臉打卡
3 關閉 Logo 燈 18:00 下班後第一位離開公司的人，請關掉公司 Logo 燈
4 檢查座位區域 如附近座位同事已離開，請確保所屬座位附近的燈光及冷氣已全部關閉
5 最後離開檢查 如為全公司最後一位下班，請再次檢查所有燈光及冷氣已關閉，並鎖上大門

注意事項

進度匯報須有明確的支撐材料，確保工作成果可追溯、可驗證。代碼提交請在 18:00 前完成，避免累積未推送的變更。18:00 準時下班，離開前請完成刷臉打卡，並關閉個人電腦及顯示屏。

規範 03 · 源代碼與版本控制

GitHub Repository 創建規範

新建 Repository 時的命名規則、建立設定與權限配置標準。

Repository 命名規則

命名格式：

Production Code - Client - Content name - Module Code

範例：

P26F-AAHK-Fire-Extinguisher-Operation-Training-A01

Module Type Code

Module Code 末尾字母代表模組類型，編號從 01 起：

代碼 類型 範例
V VR V01
M MR M01
A iPad / iOS / Exe A01
W Web-Based W01

創建設定

  • Owner 必須選擇 VTM-Digital-Limited
  • 創建時必須填寫 Description
  • Visibility 設為 Private
  • 勾選 Add README
  • 若為 Unity 項目，須添加 .gitignore 並選擇 Unity

權限設定

Repository 創建完成後，進入 Settings → Collaborators and teams 配置權限：

對象 權限
VTM Technical Department Admin
Victor Dao Admin
其餘同事 Read / Write

注意事項

請嚴格遵守命名規則與 Owner 設定，確保所有項目統一歸屬於 VTM-Digital-Limited 組織。Unity 項目務必選用 Unity 專用 .gitignore，避免提交不必要的本地與快取檔案。新建 Repository 前請先完成 GitHub 帳戶規範 中的帳戶設定。

規範 04 · 源代碼與版本控制

GitHub Desktop 提交規範

使用 GitHub Desktop 進行日常代碼提交與推送的操作標準。

帳戶註冊

  • 使用公司提供的個人郵箱進行註冊（帳戶資料設定詳見 GitHub 帳戶規範）

Commit Summary 格式

格式：

yyyy-mm-dd Content Summary...

範例：

2026-07-07 Update VTM Unity SDK

Commit Description

以 list 方式列出本次提交的具體內容，例如：

- Update VTM Unity SDK to v2.1.0
- Fix scene loading issue in Module A01
- Refactor player controller script

注意事項

請在每日 18:00 前完成當日代碼提交（參閱 每日工作流程）。Summary 須包含日期與簡要說明，Description 須清楚列出各項變更，便於追溯與審閱。SDK 版本發布時須同步遵循 版本號規範 與 CHANGELOG 規範。

規範 05 · 開發規範

版本號規範

VTM Unity SDK 採用語義化版本（SemVer 2.0）管理版本號，依變更類型遞增 MAJOR、MINOR、PATCH。

參考標準

遵循 語義化版本 2.0.0（SemVer） 規範，版本號格式為 MAJOR.MINOR.PATCH。

版本號遞增規則

序號 段名 變更場景 示例（基於 SDK 結構） 版本號變更
1 MAJOR
（主版本）
  • 核心模組（VTM_Core）不兼容 API 變更
  • 整體架構重構（如模組拆分 / 合併、目錄結構大調）
  • 底層框架（Frameworks）不兼容升級
  • VTM_Core 核心類 VTM.Core 建構函數參數變更
  • 移除 VTM_iOS 模組的 CPP 舊架構，替換為新編譯邏輯
  • 重構 Frameworks/Graphic 核心渲染邏輯
 1.0.0 → 2.0.0
2 MINOR
（次版本）
  • 新增功能模組（如新增 VTM_Android 模組）
  • 現有模組新增兼容 API / 功能（不影響舊邏輯）
  • 新增資源 / 模板（Resources / Prefabs / Template）
  • 編輯器工具（Editor 目錄）新增核心功能
  • 新增 VTM_XR 模組的 XRA 腳本與 UI 模板
  • VTM_iOS 模組新增 Frameworks/Graphic 擴充功能
  • Documents 目錄新增 SDK_Guide 多語言版本
  • VTM_Core.Editor 新增資源批量匯入工具
 1.2.3 → 1.3.0
3 PATCH
（修補版本）
  • 模組 bug 修復（不涉及 API 變更）
  • 效能優化（如 Scripts 執行效率、DLL 載入速度）
  • 文件糾錯、小範圍資源調整（Prefabs / UI 細節優化）
  • 編輯器工具（Editor）小功能修復
  • 修復 VTM_XR 模組 DLL 載入異常的 bug
  • 優化 VTM_Core 腳本執行效率，降低 CPU 佔用
  • 修正 Documents/License_Agreement 條款表述錯誤
  • 修復 VTM_iOS.Editor 打包工具的路徑識別問題
 1.3.1 → 1.3.2

注意事項

發布新版本時，須同步更新 CHANGELOG.md（詳見 CHANGELOG 規範）。若同一次發布同時包含多種變更類型，以最高優先級為準：MAJOR > MINOR > PATCH。

規範 06 · 開發規範

CHANGELOG 規範

CHANGELOG.md 記錄 VTM Unity SDK 各版本的重大變更，按版本倒序排列，並以「模組 + 目錄」分類。

撰寫原則

序號 原則 說明
1 版本倒序 最新版本置頂，依時間由新到舊排列
2 模組分類 每條變更以 【模組名】 標記所屬模組與目錄
3 變更類型 使用 Added、Changed、Fixed 三類分組
4 參考標準 基於 語義化版本控制 與 約定式提交 規範

條目格式

每條變更條目格式如下：

- 【模組名】變更描述（含目錄路徑，使用 `反引號` 標記）

範例：

- 【XR 模組】新增 `VTM_XR` 模組：含 `XRA` 腳本、`Resources/Prefabs` 基礎預製體、`UI` 模板
- 【編輯器】修復 `VTM_Core.Editor` 工具在 Unity 2021.3 版本下的相容性問題

完整範例

以下為 CHANGELOG.md 的標準結構範例：

# Changelog

所有 VTM Unity SDK 的重大變更都將記錄在此文件中。
基於 [語義化版本控制](https://semver.org/spec/v2.0.0.html) 與 [約定式提交](https://www.conventionalcommits.org/) 規範。

## [1.0.0] - 2025-12-15

### Added
- 【核心模組】初始發布 `VTM_Core` 模組：包含 `VTM.Core` 核心類、`Editor/VTM_Core.Editor` 編輯器工具及對應 DLL
- 【XR 模組】新增 `VTM_XR` 模組：含 `XRA` 腳本、`Resources/Prefabs` 基礎預製體、`UI` 模板
- 【iOS 模組】新增 `VTM_iOS` 模組：包含 `VTM.iOS` 核心邏輯、CPP 底層支援、`Frameworks/Graphic` 適配
- 【文件】新增 `Documents` 目錄：含 `Change_Log` 初始版本、`SDK_Guide` 接入文件、`License_Agreement` 授權協議
- 【框架】新增 `Frameworks` 目錄：包含 Graphic 核心框架、`Framework_Guide` 使用說明

### Changed
- 【統一配置】標準化所有模組的目錄結構（Editor/DLL/核心邏輯/Resources 分層）
- 【UI 規範】統一 `Scripts/UI` 目錄下的元件命名與介面設計

### Fixed
- 【編輯器】修復 `VTM_Core.Editor` 工具在 Unity 2021.3 版本下的相容性問題
- 【iOS 模組】解決 `VTM_iOS` 模組 `Prefabs` 載入路徑錯誤的 bug

## [0.3.0] - 2025-11-30

### Added
- 【XR 模組】新增 `VTM_XR` 模組的 `XRA` 場景模板（`Template` 目錄）
- 【編輯器】新增 `VTM_XR.Editor` 工具：支援 XR 設備參數可視化配置
- 【文件】新增 `Frameworks/Graphic` 模組的詳細使用指南（`Framework_Guide`）

### Changed
- 【效能優化】調整 `VTM_Core` 模組 `Scripts` 的執行頻率，降低效能開銷
- 【資源優化】壓縮 `VTM_iOS/Resources/Prefabs` 中的預製體體積

### Fixed
- 【XR 模組】修復 `VTM_XR` 模組 DLL 依賴缺失導致的啟動崩潰問題
- 【UI】修復 `VTM_iOS/Scripts/UI` 元件自適應佈局異常的 bug

## [0.2.0] - 2025-11-15

### Added
- 【iOS 模組】初始搭建 `VTM_iOS` 模組框架：包含 `Editor` 工具、`VTM.iOS` 核心類、CPP 底層目錄
- 【資源】新增 `VTM_iOS/Resources/Prefabs` 基礎預製體（如啟動頁、彈窗模板）
- 【腳本】新增 `VTM_iOS/Scripts/UI` 基礎元件（按鈕、輸入框、列表）

### Changed
- 【目錄規範】調整 `VTM_Core` 模組的 DLL 存放路徑，統一為 `模組名/DLL` 目錄
- 【文件】更新 `SDK_Guide`，補充 `VTM_iOS` 模組的接入流程

### Fixed
- 【核心模組】修復 `VTM_Core` 模組中 `VTM.Core` 類的空指標異常
- 【編輯器】解決 `VTM_Core.Editor` 工具匯出配置失敗的問題

## [0.1.0] - 2025-10-30

### Added
- 【核心模組】初始搭建 `VTM_Core` 模組：含 `VTM.Core` 核心邏輯、`Editor/VTM_Core.Editor` 基礎工具
- 【框架】新增 `Frameworks` 基礎目錄：包含 Graphic 核心框架雛形
- 【文件】初始建立 `Documents` 目錄：含 `Change_Log` 模板、`License_Agreement` 基礎條款

### Fixed
- 【基礎兼容】修復 `VTM_Core` 模組在 Unity 2020.3 版本下的編譯錯誤

## [0.0.1] - 2025-10-15

### Added
- 【專案初始化】建立 SDK 根目錄結構：`Assets/VTM/VTM_Unity` 核心目錄，包含模組佔位符、文件目錄雛形
- 【基礎配置】初始化 `VTM_Core` 模組的目錄結構（Editor/DLL/核心邏輯分層）

注意事項

每次發布新版本時，須在 CHANGELOG.md 頂部新增對應版本區塊，並依 版本號規範 決定版本號遞增類型。變更描述須具體到模組與目錄路徑，避免僅寫「修復 bug」等模糊表述。

規範 07

工作習慣

開發組日常工作溝通與整理習慣，持續更新補充。

習慣一覽

編號 習慣 摘要
01 已讀標記 👍 看到 Teams / Outlook 訊息後按 👍，代表已讀
02 表格整理 習慣以表格形式整理內容，表格須含序號欄
03 序號標記 回覆時以 1) 2) 3) 格式標記各項內容

01 已讀標記 👍

在看到訊息後，須按 👍 作為回應。

適用範圍 說明
Teams 訊息 閱讀後按 👍 標記已讀
Outlook 郵件 閱讀後按 👍 標記已讀
👍 代表 👍 不代表
已看到訊息 認同內容
已讀確認 喜歡內容
其他含義

02 表格整理

習慣以表格的形式整理內容，使資訊結構清晰、易於閱讀與對照。表格須包含序號欄，為每行內容標記編號。

序號 適用場景 說明
1 匯報進度 以表格列出各項任務狀態、負責人、進度等
2 整理資訊 將多項資料、對照項目以表格呈現
3 回覆說明 涉及多個欄位或條目時，優先使用表格而非長段落

03 序號標記

回覆圖表或包含多點的資訊時，習慣以序號標記各項內容，方便對方逐項閱讀與回應。序號格式為 1) 2) 3)，不使用 1. 2. 3. 格式。

序號 適用場景 說明
1 回覆圖表 針對圖表中各數據點或區塊，以 1) 2) 3) 標記回覆
2 回覆多點資訊 對方訊息含多個問題或事項時，逐項以序號回覆

範例：

1) 圖表 A 區數據已確認，與預期一致。
2) 圖表 B 區需進一步核對，預計明日回覆。
3) 關於第三點問題，建議調整為…

注意事項

工作習慣將持續補充。👍 僅作為「已讀」標記；若需進一步回覆，請以文字說明，並依習慣 03 以 1) 2) 3) 格式標記序號。
