Bug List 填寫規範

每個項目須維護一份 Bug List Excel，統一記錄測試與客戶回饋中發現的問題、優先級、負責人與審批狀態。當前表格版本為 V2.1.0，存放於 SharePoint 5 Testing 資料夾（見 管理文檔(Documents)規範）。

用途與適用時機

Bug List 是項目測試與回饋迭代的唯一正式缺陷／回饋登記表，用於：

  • 內部測試、PO 驗收、客戶試用時發現的問題與建議
  • 記錄每條回饋的優先級（是否必須修改）、負責人（PIC）與預估工時
  • 追蹤修改完成與PO / 客戶審批狀態，並以顏色標示整列進度
  • 作為 G2 / P2 等後續輪次的工作依據（見 開發階段）

口頭、Teams 私訊或未登記於 Bug List 的回饋，不視為已正式立項的修改項。

填寫流程

序號 步驟 負責方 動作
1 登記回饋 測試人員 / PO / 客戶接口人 新增一行，填寫 Stage、Log Date、Log Person、Version、Feedback Type、Comments；有材料則補 Ref. Link
2 評估優先級 PO（與相關方確認） 在「一定要改／可改可唔改／不需要改」中擇一標記 o（見下方顏色規則）
3 指派負責人與預估工時 PO / 項目負責人、各 PIC 填寫 G - PIC、P - PIC（及需要的 PO - PIC）；開工前在 PO / G / P - Cost (hr) 欄填寫各方預估工時（小時）
4 實作修改 G / P 負責人 完成後在「已修改」欄標 o；實際工時記錄於 SharePoint T&P（見 管理文檔(Documents)規範），不在 Bug List 填寫
5 PO 驗收 PO 通過標 o，不通過標 x（列變紅，須返工）
6 客戶驗收 客戶接口人 通過標 o，不通過標 x；雙方皆 o 時列變深綠，正式結案

主表欄位說明

序號 欄位 填寫規則
1 Index 序號，一條回饋對應一行
2 Stage 發現問題時所處的開發階段，如 PP、P1、G2（見 開發階段）
3 Log Date 登記日期，格式 yyyy-mm-dd
4 Log Person 登記人；從下拉選單選取（團隊成員、Client、Consultant）
5 Version 發現問題時的軟件版本號（如 0.39.1、1.0.0），須與實際測試包一致（見 版本號規範）
6 Feedback Type 回饋類別，從下拉選單選取：1. 設計、2. 美術、3. 技術、4. 功能、5. 其他
7 Comments / Feedback 問題或建議的具體描述，須可重現、可理解；避免僅寫「有 bug」等模糊表述
8 Refer 關聯其他條目的序號（如重複回饋、拆分項之間的引用）
9 Ref. Link 相關截圖、影片、Planner 任務、SharePoint 文件等連結
10 一定要改 / 可改可唔改 / 不需要改 優先級三選一：在對應欄位填入 o（見下方「優先級與顏色規則」）
11 PO - PIC / G - PIC / P - PIC PO、美術、程式各方的負責人；依 Feedback Type 與實際分工填寫
12 PO - Cost (hr) / G - Cost (hr) / P - Cost (hr) 各方處理該條目的預估工時（小時），須在開始工作前填寫，用於排期與資源評估；非實際工時
13 已修改 開發方已完成修改時填入 o
14 PO Approved PO 驗收：o 通過、x 不通過（需返工）
15 Client Approved 客戶驗收：o 通過、x 不通過（需返工）

優先級與顏色規則

「一定要改」「可改可唔改」「不需要改」三欄每次只能標記一個 o。標記後，該列背景或文字顏色會依條件格式自動變化。

序號 優先級 標記方式 顏色效果 含義
1 一定要改 在「一定要改」欄填入 o 預設白底（標準列） 必須在本輪或約定時限內修正；未完成前保持活躍追蹤
2 可改可唔改 在「可改可唔改」欄填入 o 整列灰色背景 建議改進項；視工時與排期決定是否納入本輪，不阻塞主線交付
3 不需要改 在「不需要改」欄填入 o 文字變淡／灰色字 經評估不予修改；該列視為關閉，不再追蹤

修改與審批狀態顏色規則

右側「已修改」「PO Approved」「Client Approved」使用 o（是／通過）、x（驗收不通過）或空白（待處理）。填寫後觸發下列整列顏色：

序號 條件 顏色效果 狀態說明
1 PO Approved 或 Client Approved 為 x 整列紅色背景 驗收不通過，須返工後重新標記「已修改」並再次提交審批
2 已修改 o，且 PO Approved o，且 Client Approved o 整列深綠色背景 修改完成，PO 與客戶均已確認 — 正式結案
3 已修改 o，且 PO Approved o，Client Approved 為空白 整列淺綠色背景 開發與 PO 驗收已通過，待客戶確認
4 已修改 o，審批欄尚未全部通過（且無 x） 維持白底或依表內其他規則 已提交修改，待 PO 或客戶驗收
5 尚未標記「已修改」 依優先級規則（白底／灰底／淡字） 進行中或待處理

顏色由 Excel 條件格式自動套用，請勿手動改背景色；若顏色與符號不一致，應檢查是否填錯欄位或遺漏 o / x。

注意事項

  • PO / G / P - Cost (hr) 為預估工時，開工前填寫；實際花費的工時須記錄於 SharePoint T&P（0 Project Management），見 每日工作流程
  • 表格版本升級（如 V2.1.0 → 下一版）時，須由項目負責人統一更換模板，勿自行增刪欄位以免條件格式失效
  • Version 欄須對應實際測試的安裝包版本，方便回溯問題所在 commit / 發布
  • 同一問題若跨美術與程式，可在一條 Comments 中說明，或拆成多行並以 Refer 關聯
  • 標為「不需要改」的項目不再進入 G2 / P2 排期；標為「一定要改」且未結案者，應同步反映於 Planner 任務（見 管理任務(Task)規範）
  • 客戶試用或 UAT 階段的新回饋，Log Person 選 Client，並在 Comments 中保留可重現步驟
