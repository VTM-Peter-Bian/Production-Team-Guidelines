版本號規範

所有需發布版本的交付物均採用語義化版本（SemVer 2.0）管理，依變更類型遞增 MAJOR、MINOR、PATCH。

適用範圍

本規範適用於開發組所有具備對外發布或交付的版本號管理場景，不限於單一技術棧：

序號 交付物類型 版本號示例 說明
1 Unity SDK / 共用框架 VTM Unity SDK v2.1.0 跨項目複用的核心 SDK、插件、框架庫
2 培訓項目 / 客戶端應用 P26F-AAHK-…-A01 v1.0.0 VR / MR / iOS / Web 等交付給客戶的培訓應用；首次交付客戶固定為 v1.0.0（見下方「客戶交付版本」）
3 內部工具 / 編輯器擴充 BuildTool v0.4.1 打包工具、資源匯入工具、自動化腳本等
4 文件與配置包 SDK_Guide v1.0.0 接入指南、授權協議、配置模板等可獨立版本化的資產

客戶交付版本

交付給客戶的安裝包或執行檔，首次正式交付統一使用 v1.0.0，後續變更再依語義化版本規則遞增。

序號 場景 版本號 說明
1 首次交付客戶 1.0.0 不論內部開發歷經多少迭代，正式交予客戶的 APK、EXE、安裝包等均以 v1.0.0 發布
2 客戶端 Bug 修復 1.0.0 → 1.0.1 修復缺陷、效能優化、小範圍資源調整等，遞增 PATCH
3 客戶端功能擴充 1.0.1 → 1.1.0 新增培訓模組、場景或兼容性功能，遞增 MINOR
4 客戶端重大變更 1.x.x → 2.0.0 不兼容的架構重構、核心流程改版等，遞增 MAJOR

範例：某 VR 培訓項目首次交付客戶 Fire-Training.apk v1.0.0；客戶反饋後修復載入問題，發布 v1.0.1；後續新增一個培訓關卡，發布 v1.1.0。

內部開發階段（尚未交付客戶）的版本號管理不受此限制，可依項目需要從 0.x.x 起步；以首次正式交付客戶的版本為 1.0.0 起點。

參考標準

遵循 語義化版本 2.0.0（SemVer） 規範，版本號格式為 MAJOR.MINOR.PATCH。

版本號遞增規則

序號 段名 變更場景（通用） 版本號變更
1 MAJOR
（主版本）
  • 對外 API / 公共接口發生不兼容變更
  • 整體架構重構（模組拆分 / 合併、目錄結構大調）
  • 底層框架或核心依賴的不兼容升級
  • 舊版本用戶無法在不修改代碼的情況下升級
 1.0.0 → 2.0.0
2 MINOR
（次版本）
  • 新增功能模組或業務場景（向後兼容）
  • 現有模組新增兼容 API / 功能（不影響舊邏輯）
  • 新增資源、模板、配置項
  • 工具類新增核心功能
 1.2.3 → 1.3.0
3 PATCH
（修補版本）
  • Bug 修復（不涉及 API 或行為變更）
  • 效能優化、載入速度改善
  • 文件糾錯、小範圍資源或 UI 細節調整
  • 工具類小功能修復
 1.3.1 → 1.3.2

參考示例

以下以 VTM Unity SDK 為例，說明上述通用規則在具體項目中的應用。其他交付物類型應依相同原則，替換為對應的模組名、目錄或功能範圍。

序號 段名 示例（VTM Unity SDK）
1 MAJOR
  • VTM_Core 核心類 VTM.Core 建構函數參數變更
  • 移除 VTM_iOS 模組 CPP 舊架構，替換為新編譯邏輯
  • 重構 Frameworks/Graphic 核心渲染邏輯
2 MINOR
  • 新增 VTM_XR 模組的 XRA 腳本與 UI 模板
  • VTM_iOS 模組新增 Frameworks/Graphic 擴充功能
  • Documents 目錄新增 SDK_Guide 多語言版本
3 PATCH
  • 修復 VTM_XR 模組 DLL 載入異常
  • 優化 VTM_Core 腳本執行效率
  • 修正 Documents/License_Agreement 條款表述錯誤

注意事項

發布新版本時，須同步更新 CHANGELOG.md（詳見 CHANGELOG 規範）。若同一次發布同時包含多種變更類型，以最高優先級為準：MAJOR > MINOR > PATCH。培訓項目可在 Repository 名稱之外，另以 Tag 或 Release 標記語義化版本號。交付客戶的 APK、EXE 等安裝包，首次交付須為 v1.0.0，後續更新依上方「客戶交付版本」規則遞增。
