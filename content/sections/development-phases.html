術語定義

開發階段（PP / G / P）

開發組項目從前置準備到正式製作、再到客戶回饋迭代的階段劃分。Pre-Production 縮寫為 PP，在正式開發之前；G 代表 Graphic Team 美術輪次；P 代表 Program Team 程式輪次；數字 1 / 2 / 3 代表第幾輪。

為什麼要劃分這些階段

開發組並非「一開始就同時全面開做美術與程式」，而是先把研究、設計與風險驗證做完，再進入正式製作；正式交付可運行軟件後，再依經評估確認的客戶 feedback 進入下一輪修改。階段劃分的目的是：

  • 讓團隊清楚現在該做什麼、不該做什麼（例如：flow 與關鍵技術尚未驗證前，不應直接進入 G1 / P1 正式製作）
  • 讓 G 與 P 的完成標準可檢查、可匯報（對應 完成（Completion））
  • 讓 Planner 任務命名中的 PP、G1、P1 等標記有統一含義（見 創建任務(Task)規範）

飯店比喻（理解用）

以飯店出餐比喻開發節奏：

序號 飯店情境 對應階段 說明
1 廚師確認廚房裡的工具、材料是否足夠，以及固定菜單上的菜要怎麼做 PP（Pre-Production） 客人尚未正式點單開炒前的準備；研究、設計、試跑、R&D 都在此完成。完成後可提供參考圖／影片／APK／EXE 等供客戶確認
2 接到客人點單後，正式開始炒菜 G1 / P1 正式製作開始：美術把資源備齊（G1），程式按既定 flow 與資源把菜炒出來（P1）
3 因我們做錯菜而返工（例如：菜單原定甜味卻做成鹹味、上錯菜、漏配菜等） G2 / P2 客戶收到可運行軟件後，發現我們做錯、做漏或與已確認規格不符而修正的輪次——不是客人主觀說「不喜歡這個口味」，也不是隨意換新菜單
4 少數情況再改第二輪、第三輪 G3 / P3 預計較少進入；主要處理嚴重 bug，而非一般性調整

階段總覽與理想順序

理想情況下的推進順序如下（實際項目可因資源到位情況略有重疊，但完成定義不變）：

PP → G1 → P1 →（客戶收到可運行軟件）→ G2 / P2 →（較少）G3 / P3

序號 階段 負責團隊 時機 完成定義（一句話）
1 PP（Pre-Production） Graphic / Program（及相關職能） G1、P1 正式開始之前 正式開發前的研究、設計與前置工作已就緒
2 G1 Graphic Team 正式開始美術製作（含內部領導與客戶審核，及依 feedback 修改） 該項目所有相關美術資源已全部準備就緒，且已走完內部領導與客戶審核流程
3 P1 Program Team 理想情況下於 G1 完成後開始 預定流程（flow）已全部實現；客戶可收到可運行軟件
4 G2 Graphic Team G1 / P1 交付後，經評估確認應修正的客戶 feedback 本輪已確認的美術資源修改已完成
5 P2 Program Team G1 / P1 交付後，經評估確認應修正的客戶 feedback 本輪已確認的程式問題與相關調整已完成
6 G3 Graphic Team 預計較少；主要因嚴重美術問題或 bug 第三輪已確認的修改已完成
7 P3 Program Team 預計較少；主要因嚴重程式 bug 第三輪已確認的修改已完成

PP = Pre-Production（前置製作）；G = Graphics（美術）；P = Programming（程式）。同一數字輪次中，G 與 P 可依 feedback 內容並行，但各自有獨立完成定義。

01 PP — Pre-Production（前置製作）

PP（Pre-Production） 位於 G1 與 P1 之前。凡「正式開始開發前」應處理的工作，都應歸入本階段，而不是混進 G1 / P1 的正式製作工時。

階段目的

在正式開炒（G1 / P1）之前，把流程、技術風險與製作路徑想清楚、驗證清楚，避免正式開發中途才發現 flow 不通、技術不可行或資源規格未定。以飯店比喻：廚師須先確認廚房裡的工具是否齊全、材料是否足夠、固定要做的菜怎麼做出來——以上都須在本階段釐清並驗證。

PP 完成後，通常已可提供參考圖、影片、APK、EXE等交付物，供客戶確認方向與規格，再進入 G1 / P1 正式製作。當團隊累積了 Design Library、UI Library、Function Library 後，本階段相對會更簡單、更快完成。

典型工作內容

序號 類別 說明 示例
1 研究與設計 flow 釐清培訓／產品流程，形成可製作的設計依據 流程拆解、互動設計、與客戶確認關鍵路徑（可結合 TSA）
2 Virtual run 用較低成本把流程「跑一遍」，驗證邏輯與體驗是否成立 紙面／簡模走查、流程演示、關鍵節點試跑
3 R&D 針對不確定技術或新功能做預研與驗證 新 SDK、特殊互動、效能／硬體限制驗證、原型功能
4 工具、材料與製作路徑確認 確認開發所需工具鏈、素材／資源是否足夠，以及固定功能／流程要怎麼實現 引擎與插件版本、硬體限制、素材規格、標準模組能否複用（Design / UI / Function Library）
5 其他前置工作 任何為正式製作掃清障礙、且應在開做前完成的事項 規格對齊、風險清單、參考圖／影片／APK／EXE 供客戶確認

進入與完成

序號 項目 說明
1 進入條件 項目已具備足夠的需求／目標輸入，可以開始研究與設計（尚未進入正式美術／程式量產）
2 完成定義 正式開發所需的研究、設計與前置驗證已就緒；工具、材料與製作路徑已確認；必要時已提供參考圖／影片／APK／EXE 等供客戶確認，團隊可以有依據地進入 G1 / P1
3 不應算作本階段完成的情況 僅完成部分研究、flow 仍未定案、關鍵 R&D 未驗證通過，卻已開始把正式資源／正式功能當最終交付推進

02 G1 / P1（正式製作）

這是「接到點單後正式開炒」的階段。PP 解決的是「做什麼、怎麼做是否可行」；G1 / P1 解決的是「按既定方案把可交付成果做出來」。

G1（Graphic Team 第一輪）

序號 項目 說明
1 含義 Graphic Team 正式開始美術製作工作
2 工作焦點 依已確認的設計／規格，產出項目所需的美術資源（模型、貼圖、UI、場景、動畫等，以項目實際範圍為準）
3 完成定義 該項目所有關於美術的資源已經全部準備就緒——不是「做了一部分」，而是本輪預定範圍內的美術資源已齊備、已走完內部領導與客戶審核並處理完本輪應修正的 feedback，可供程式按 flow 使用
4 與 P1 的關係 G1 的產出是 P1 理想開工條件之一：Program Team 應能依據已有 flow 與 Graphic 資源推進製作
5 內部與客戶審核 Graphic 產出美術資源後，須依次經內部領導審核與客戶審核；兩方皆可能提出 feedback 而需修改。規劃 G1 工時時須預留內部與客戶審核及修改的時間——開發組目前常見問題是未預留這段緩衝，導致排期被動壓縮

P1（Program Team 第一輪）

序號 項目 說明
1 含義 Program Team 依既定 flow 與 Graphic 資源，正式實現軟件功能與流程
2 理想開工時機 G1 完成之後開始：此時 flow 已定、美術資源已就緒，可減少「邊等資源邊改架構」的返工
3 工作焦點 按已有 flow 把預定流程全部做出來，整合 Graphic 資源，形成客戶可運行的軟件
4 完成定義 預定的流程已全部實現。此時客戶應已能收到可以運行的軟件（第一輪可運行交付），而不是僅有局部 demo 或無法串起完整預定 flow 的半成品

G1 與 P1 完成後代表什麼

  • 美術側：本輪所需資源已齊，且已走完內部領導與客戶審核（G1 完成）
  • 程式側：預定 flow 已全部落地（P1 完成）
  • 對客戶：已具備「可運行軟件」作為第一輪交付與收集 feedback 的基礎

03 G2 / P2（第一輪回饋迭代）

客戶在收到可運行軟件後，若發現我們做錯、做漏、品質有問題或與已確認規格不符，才進入 G2 與／或 P2 修正。這些回饋可能落在美術、程式，或兩者都有。

重要：此處的 feedback 指的是確實是我們做錯、做漏或與已確認規格不符之處——我們應負責修正。客戶提出的每一項意見，並不會自動進入 G2 / P2；須先經團隊評估，確認納入本輪修改範圍後，才開始計入 G2 / P2 工時。

是否進入 G2 / P2 的評估因素

序號 評估因素 說明
1 是否確實是我們的問題 屬於我們做錯、做漏、與 T&P／已確認規格不符，還是客戶事後追加的新需求
2 需求是否合理 修改方向是否符合項目目標與已確認範圍；不合理的改動需求不應納入本輪
3 技術上是否可行 在現有架構、工具鏈與硬體限制下能否實現
4 是否有時間／工時 項目排期與 man-day 是否容許納入；超出範圍者須另行申請或協商

綜合以上因素確認應修正的項目後，形成本輪已確認的 feedback 清單，再進入 G2 / P2 執行。客戶提出但不合理的改動需求，不應因「客戶有提」就自動修改。

序號 階段 常見觸發 完成定義
1 G2 經評估確認應修正的美術問題（造型、材質、UI、場景細節、動畫表現等） 本輪已確認的美術資源修改已完成
2 P2 經評估確認應修正的程式 bug、流程偏差、互動問題、穩定性問題等 本輪已確認的程式問題與相關調整已完成

使用原則

  • G2 / P2 的前提通常是：客戶已基於 G1 / P1 的可運行成果提出具體 feedback，且團隊已完成上述評估、形成確認清單
  • 同一輪中，若已確認的 feedback 同時包含美術與程式，G2 與 P2 可並行，但須各自對齊完成定義
  • 本輪完成，指的是本輪已確認的 feedback 清單做完，而不是「客戶永遠不再提意見」，也不是「客戶提過的都改完」

04 G3 / P3（後續回饋迭代）

進入 G3、P3 的情況在規劃上預計較少。與 G2 / P2 不同，本輪主要處理嚴重 bug——即影響正常使用、導致流程中斷、或造成明顯錯誤結果的問題；一般性調整或新需求不應以此階段為預設入口。

序號 階段 常見觸發 完成定義
1 G3 上一輪後仍存在的嚴重美術問題（如關鍵資源錯誤、嚴重穿幫影響培訓效果等，較少發生） 第三輪已確認的修改已完成
2 P3 上一輪後仍存在的嚴重程式 bug（如崩潰、流程卡死、關鍵功能失效等，較少發生） 第三輪已確認的修改已完成

若項目出現超過三輪的持續迭代，仍沿用「PP / G / P + 輪次數字」的命名邏輯溝通，但應同步檢視需求是否反覆變更、PP / G1 / P1 的完成定義是否當初未真正達標，以及 G2 / P2 的 feedback 評估是否到位。

G 與 P 如何對照理解

輪次 Graphic（G） Program（P） 對客戶的意義
1 正式產出並備齊美術資源 按預定 flow 全部實現 首次可運行軟件交付
2 依已確認的 feedback 改美術 依已確認的 feedback 修程式／流程 第一輪回饋後的修正版（非客戶提過即改）
3 嚴重美術問題修正（較少） 嚴重程式 bug 修正（較少） 後續針對嚴重問題的再修正

注意事項

  • 任一階段標記為「完成」時，須符合 完成（Completion）：目標達成、質量達標、可追溯，而不是「做過一部分」。
  • Planner 任務名稱中的 PP、G1、P1、P2 等，即對應本頁階段（範例見 創建任務(Task)規範）。
  • 理想順序是 PP → G1 → P1；若因排期必須部分重疊，仍不得降低各階段的完成定義。
  • G1 規劃時須預留美術資源內部領導與客戶審核、以及 feedback 修改的工時，不可只按「純製作」估算。
  • G2 / P2 主要處理我們做錯、做漏或品質不符之處；須先評估是否合理、技術可行、工時足夠，確認後才納入本輪；G3 / P3 主要處理嚴重 bug，預計較少進入。
