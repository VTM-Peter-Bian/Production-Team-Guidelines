術語定義

開發階段（Pre-Production / G / P）

開發組項目從前置準備到正式製作、再到客戶回饋迭代的階段劃分。Pre-Production 在正式開發之前；G 代表 Graphic Team 美術輪次；P 代表 Program Team 程式輪次；數字 1 / 2 / 3 代表第幾輪。

為什麼要劃分這些階段

開發組並非「一開始就同時全面開做美術與程式」，而是先把研究、設計與風險驗證做完，再進入正式製作；正式交付可運行軟件後，再依客戶 feedback 進入下一輪修改。階段劃分的目的是：

  • 讓團隊清楚現在該做什麼、不該做什麼（例如：flow 與關鍵技術尚未驗證前，不應直接進入 G1 / P1 正式製作）
  • 讓 G 與 P 的完成標準可檢查、可匯報（對應 完成（Completion））
  • 讓 Planner 任務命名中的 G1、P1 等標記有統一含義（見 創建任務規範）

飯店比喻（理解用）

以飯店出餐比喻開發節奏：

序號 飯店情境 對應階段 說明
1 備料、設計菜單、試做、確認廚具與流程 Pre-Production 客人尚未正式點單開炒前的準備；研究、設計、試跑、R&D 都在此完成
2 接到客人點單後，正式開始炒菜 G1 / P1 正式製作開始：美術把資源備齊（G1），程式按既定 flow 與資源把菜炒出來（P1）
3 客人吃到第一道菜後提出調整（口味、擺盤、漏項） G2 / P2 客戶已收到可運行軟件後的 feedback 修改輪次
4 少數情況再改第二輪、第三輪 G3 / P3 概念同 G2 / P2；預計較少進入，但仍可能發生

階段總覽與理想順序

理想情況下的推進順序如下（實際項目可因資源到位情況略有重疊，但完成定義不變）：

Pre-Production → G1 → P1 →（客戶收到可運行軟件）→ G2 / P2 →（較少）G3 / P3

序號 階段 負責團隊 時機 完成定義（一句話）
1 Pre-Production Graphic / Program（及相關職能） G1、P1 正式開始之前 正式開發前的研究、設計與前置工作已就緒
2 G1 Graphic Team 正式開始美術製作 該項目所有相關美術資源已全部準備就緒
3 P1 Program Team 理想情況下於 G1 完成後開始 預定流程（flow）已全部實現；客戶可收到可運行軟件
4 G2 Graphic Team G1 / P1 交付後，依客戶 feedback 本輪客戶提出的美術資源修改已完成
5 P2 Program Team G1 / P1 交付後，依客戶 feedback 本輪客戶提出的程式問題與相關調整已完成
6 G3 Graphic Team 預計較少；概念同 G2 第三輪美術修改已完成
7 P3 Program Team 預計較少；概念同 P2 第三輪程式修改已完成

G = Graphics（美術）；P = Programming（程式）。同一數字輪次中，G 與 P 可依 feedback 內容並行，但各自有獨立完成定義。

01 Pre-Production（前置製作）

Pre-Production 位於 G1 與 P1 之前。凡「正式開始開發前」應處理的工作，都應歸入本階段，而不是混進 G1 / P1 的正式製作工時。

階段目的

在正式開炒（G1 / P1）之前，把流程、技術風險與製作路徑想清楚、驗證清楚，避免正式開發中途才發現 flow 不通、技術不可行或資源規格未定。

典型工作內容

序號 類別 說明 示例
1 研究與設計 flow 釐清培訓／產品流程，形成可製作的設計依據 流程拆解、互動設計、與客戶確認關鍵路徑（可結合 TSA）
2 Virtual run 用較低成本把流程「跑一遍」，驗證邏輯與體驗是否成立 紙面／簡模走查、流程演示、關鍵節點試跑
3 R&D 針對不確定技術或新功能做預研與驗證 新 SDK、特殊互動、效能／硬體限制驗證、原型功能
4 其他前置工作 任何為正式製作掃清障礙、且應在開做前完成的事項 規格對齊、工具鏈準備、素材規格定義、風險清單

進入與完成

序號 項目 說明
1 進入條件 項目已具備足夠的需求／目標輸入，可以開始研究與設計（尚未進入正式美術／程式量產）
2 完成定義 正式開發所需的研究、設計與前置驗證已就緒，團隊可以有依據地進入 G1 / P1
3 不應算作本階段完成的情況 僅完成部分研究、flow 仍未定案、關鍵 R&D 未驗證通過，卻已開始把正式資源／正式功能當最終交付推進

02 G1 / P1（正式製作）

這是「接到點單後正式開炒」的階段。Pre-Production 解決的是「做什麼、怎麼做是否可行」；G1 / P1 解決的是「按既定方案把可交付成果做出來」。

G1（Graphic Team 第一輪）

序號 項目 說明
1 含義 Graphic Team 正式開始美術製作工作
2 工作焦點 依已確認的設計／規格，產出項目所需的美術資源（模型、貼圖、UI、場景、動畫等，以項目實際範圍為準）
3 完成定義 該項目所有關於美術的資源已經全部準備就緒——不是「做了一部分」，而是本輪預定範圍內的美術資源已齊備、可供程式按 flow 使用
4 與 P1 的關係 G1 的產出是 P1 理想開工條件之一：Program Team 應能依據已有 flow 與 Graphic 資源推進製作

P1（Program Team 第一輪）

序號 項目 說明
1 含義 Program Team 依既定 flow 與 Graphic 資源，正式實現軟件功能與流程
2 理想開工時機 G1 完成之後開始：此時 flow 已定、美術資源已就緒，可減少「邊等資源邊改架構」的返工
3 工作焦點 按已有 flow 把預定流程全部做出來，整合 Graphic 資源，形成客戶可運行的軟件
4 完成定義 預定的流程已全部實現。此時客戶應已能收到可以運行的軟件（第一輪可運行交付），而不是僅有局部 demo 或無法串起完整預定 flow 的半成品

G1 與 P1 完成後代表什麼

  • 美術側：本輪所需資源已齊（G1 完成）
  • 程式側：預定 flow 已全部落地（P1 完成）
  • 對客戶：已具備「可運行軟件」作為第一輪交付與收集 feedback 的基礎

03 G2 / P2（第一輪回饋迭代）

依過往經驗，客戶在收到可運行軟件後，往往會再提出 feedback。這些回饋可能落在美術、程式，或兩者都有，因此分別進入 G2 與／或 P2。

序號 階段 常見觸發 完成定義
1 G2 客戶要求修改美術資源（造型、材質、UI、場景細節、動畫表現等） 本輪客戶提出的美術資源修改已完成
2 P2 客戶回報程式 bug、流程調整、互動修正、穩定性問題等 本輪客戶提出的程式問題與相關調整已完成

使用原則

  • G2 / P2 的前提通常是：客戶已基於 G1 / P1 的可運行成果提出具體 feedback
  • 同一輪中，若 feedback 同時包含美術與程式，G2 與 P2 可並行，但須各自對齊完成定義
  • 本輪完成，指的是本輪已確認的 feedback 清單做完，而不是「客戶永遠不再提意見」

04 G3 / P3（後續回饋迭代）

進入 G3、P3 的情況在規劃上預計較少，但概念與 G2 / P2 相同：客戶在上一輪修改後再次提出新的 feedback，美術進入下一輪修改（G3），程式進入下一輪修改（P3）。

序號 階段 說明 完成定義
1 G3 第三輪美術修改（較少發生） 第三輪美術修改已完成
2 P3 第三輪程式修改（較少發生） 第三輪程式修改已完成

若項目出現超過三輪的持續迭代，仍沿用「G / P + 輪次數字」的命名邏輯溝通，但應同步檢視需求是否反覆變更、Pre-Production / G1 / P1 的完成定義是否當初未真正達標。

G 與 P 如何對照理解

輪次 Graphic（G） Program（P） 對客戶的意義
1 正式產出並備齊美術資源 按預定 flow 全部實現 首次可運行軟件交付
2 依 feedback 改美術 依 feedback 修程式／流程 第一輪回饋後的修正版
3 再一輪美術修改（較少） 再一輪程式修改（較少） 後續回饋後的再修正

注意事項

  • 任一階段標記為「完成」時，須符合 完成（Completion）：目標達成、質量達標、可追溯，而不是「做過一部分」。
  • Planner 任務名稱中的 G1、P1、P2 等，即對應本頁階段（範例見 創建任務規範）。
  • 理想順序是 Pre-Production → G1 → P1；若因排期必須部分重疊，仍不得降低各階段的完成定義。
